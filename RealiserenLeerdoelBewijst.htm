Zelf reflectie op feedback
Ik heb een aantal screenshot gedeeld waar ik vind dat er belangrijke feedback wordt benoemd. Ik heb nog veel meer feedback gekregen maar ik heb voor deze selectie gekozen omdat het mij het meest heeft geleerd in het behalen van deze leerdoel.
De eerste paar feedbackpunten gingen niet over de code maar mijn werkwijze. Ik heb het besloten toe te voegen als feedback omdat om een feature te realiseren een duidelijke structuur en overzicht van de code belangrijk is. Bij het cre�ren van een nieuwe branch is er een stramien dat gevolgd moet worden voor de benaming ervan. Ik heb dit niet gedaan en bedacht er maar een die logisch klonk en dat was fout. Het bedrijf heeft een duidelijk stramien voor de benaming. De benaming moet als volgt gaan eerst leg je uit wat je toevoegt dit kan een feature of een fix zijn. Vervolgens zeg je wat de naam is van de feature of fix en als je op een platform werkt eindig je de naam met de platformzijde Android of IOS. En je scheidt al deze stukken met een �\� en voor spaties gebruik je een �-�. Dus een voorbeeld is Feature/areas-of-activity/Android.
Vervolgens zoals te zien in figuur 3 moet ik meer comments zetten bij de variabelen om mijn code overzichtelijker te maken.
Verder kreeg ik ook te horen dat mijn commit berichten duidelijke informatie moesten bevatten over de commit, figuur 4. Wat inhoud dat ik uitleg wat voor verandering de commit in de code heeft gedaan. Het schrijven van een commit bericht heeft ook stramien dat het bedrijf volgt, zoals: beginnen met een hoofdletter, schrijven in de gebiedende wijs, minder dan 50 karakters enzovoorts. Meer is te lezen op de link: https://chris.beams.io/posts/git-commit/
Ik heb deze drie feedback punten meteen ge�mplementeerd in mijn werk. Ik heb de naam van mijn branch aangepast aan de juiste benaming toegepast.
